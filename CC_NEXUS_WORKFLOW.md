# CC 的 Nexus 开发工作流

> 不是给 Nexus 用的工具链——是 CC 自己每次操作 Nexus 必须遵守的流程。
> 目的：防止跑偏、防止忘记大局、防止交付半成品。

---

## 六阶段工作流

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1.架构   │───→│ 2.执行   │───→│ 3.验证   │───→│ 4.运行   │───→│ 5.调试   │───→│ 6.交付   │
│   设计   │    │          │    │          │    │          │    │  (按需)  │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     ↓               ↓               ↓               ↓                              ↓
  画全貌图       上下游一起看    31项测试跑通    重启看日志                     提交+日记+清理
```

---

## 阶段1: 架构设计 — "不动代码"

**什么时候进入**：接到新任务时

**做什么**：
1. 先回答三个问题：
   - 这个改动的**上下游**是谁？（谁调用它，它调用谁）
   - 这个改动的**影响范围**多大？（vulture + 依赖图确认）
   - 这个改动**完成后怎么验证**？（什么日志、什么指标算成功）
2. 画出全貌图再动手
3. 列出改动清单（文件×N，每个文件的改动点）

**输出**：依赖图 + 影响范围 + 改动清单

**检查点**：
- [ ] 三个问题都有答案
- [ ] 改动清单写好了
- [ ] 跟张凯确认了方向再动手

---

## 阶段2: 执行 — "一根线穿到底"

**什么时候进入**：架构设计完成后

**做什么**：
1. 按改动清单逐个文件修改
2. 每次改完一个文件，检查：
   - 调用方是否需要同步修改？（搜索 `grep -rn "func_name"` 全库确认）
   - 参数签名是否向后兼容？（如果改签名，必须搜所有调用方）
3. 改代码同时写注释：为什么改、怎么改的
4. 每改完 3 个文件，停下来编译验证一次

**输出**：修改后的代码 + 调用方同步修改

**检查点**：
- [ ] 每个文件的调用方都检查过
- [ ] 参数兼容性确认过
- [ ] 编译通过（不是"应该能过"，是真跑过）

---

## 阶段3: 验证 — "31 项测试"

**什么时候进入**：所有代码修改完成后

**做什么**：
1. 编译检查：每个修改过的文件 `py_compile.compile()`
2. 导入检查：每个修改过的模块 `__import__()`
3. 单元测试：针对修改的 API 写测试（如 `experience_bank.add_content(tags=...)`）
4. 集成测试：端到端链路（如 knowledge_trainer → experience_bank → training_executor）
5. 工具扫描：`vulture` 确认没引入新死代码，`grep` 确认没留 `.bak` 文件

**输出**：测试通过截图 + 测试覆盖范围

**检查点**：
- [ ] 所有修改文件编译通过
- [ ] 所有修改模块可导入
- [ ] 单元测试通过
- [ ] 端到端链路测试通过
- [ ] vulture 无新增问题

---

## 阶段4: 运行 — "看日志"

**什么时候进入**：所有测试通过后

**做什么**：
1. 重启 Nexus
2. 等 2 分钟，看日志：
   - `grep ERROR` — 必须为 0
   - `grep WARNING` — 数量是否合理
   - 改动相关的模块有没有新日志出现
3. 等 5 分钟，看关键指标：
   - SignalTracker scores 是否正常
   - 知识管线是否有产出
   - DeepSeek/MiniMax 调用是否正常

**输出**：运行日志截图 + 关键指标

**检查点**：
- [ ] 重启后 ERROR = 0
- [ ] 改动的模块有正常日志输出
- [ ] 关键指标无异常

---

## 阶段5: 调试（按需）— "先停下来"

**什么时候进入**：验证或运行阶段发现新问题

**做什么**：
1. **先停下来**，不要继续改
2. 回到阶段 1：重新画图、重新理解
3. 搞不明白就问张凯

**检查点**：
- [ ] 搞明白了才继续
- [ ] 搞不明白就问，不自己瞎跑

---

## 阶段6: 交付 — "全落盘"

**什么时候进入**：运行验证通过后

**做什么**：
1. Git 提交：
   - `git add` 所有修改文件 + 新文件
   - `git rm` 确认可以删的文件
   - `git commit -m "vX.Y: 描述(根因→修复→验证)"`
2. 清理：
   - 删除临时文件（`.bak`、`.cc-backup`、`C:\...\workspace\test*.py`）
   - 确认无遗漏的调试代码
3. 文档更新：
   - `NEXUS_DIARY_2026-XX-XX.md` — 写**日记**（不是 changelog）
     - 必须有叙事：发现问题→分析→修复→心得
     - 必须有数字：修改文件数、行数、测试通过数
     - 必须有教训：今天犯什么错了、明天要注意什么
   - `nexus_daemon.bat` — 版本号更新
   - `MEMORY.md` — 如需要，更新长期记忆
4. 最终确认：
   - 工作区 clean（`git status`）
   - `.env` 配置已落盘
   - Nexus 运行正常

**输出**：Git commit + 日记 + 干净的 workspace

**检查点**：
- [ ] 所有修改已 `git commit`
- [ ] 临时文件已清理
- [ ] 日记已写（叙事风格，不是 changelog）
- [ ] 启动脚本版本号已更新
- [ ] 工作区 `git status` clean
- [ ] 跟张凯确认"还有需要做的吗"
