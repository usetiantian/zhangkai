## 追问为什么（核心方法论）

**追问为什么**：不是问一次就停，是追问因果链。

**表象 → 为什么？ → 直接原因 → 为什么？ → 根因 → 为什么？ → 系统性原因**

遇到问题必须追问这个链条：
1. 第1层：看到了什么现象？（表象）
2. 第2层：为什么会出现这个现象？（直接原因）
3. 第3层：为什么会出现这个直接原因？（根因）
4. 第4层：为什么系统会有这个根因缺陷？（系统性原因）
5. 第5层：这个根因还会影响其他什么地方？（举一反三）
6. 第6层：解决方案是修复这一层还是上一层？（优先级）

**反面案例**：RPC返回200就认为正常，没追问为什么7个collection同时报错
**正面案例**：追问到ChromaDB多线程共享client的并发安全问题

**新增：静默代码检查法（2026-04-10）**
- 遇到任何"看起来在工作但没反馈"的现象 → 追问why链
- 静默代码三特征：无返回值、无日志、无结构化结果
- 修复模式：静默 → 有日志 + 结构化返回值
- 适用场景：定时任务跳过、执行路径有分支但不分 outcome
- 举一反三：learn_ai_from_web / daily_light_learning 等也有同类静默返回问题

### 核心：两条思维路径

**路径A：修复性解决**（问题在系统内部，修改代码）
当问题的根因在现有系统的代码/逻辑中时，用这条路。
典型场景：Bug、逻辑错误、参数不对、代码重复定义

**路径B：架构性解决**（问题根因在系统间的触发机制，用修复代价高或修复不了时）
当问题的根因是"外部系统的触发机制"或"系统设计本身"时，用这条路。
典型场景：系统A的正常行为触发系统B的异常、跨系统循环依赖、定时任务副作用
核心原则：**消灭触发源比修复后果更根本**

---

### 路径A：修复性问题解决链

**触发条件**：任何代码修改/修复/Bug任务，或"这个问题很简单直接改一下"的想法（越是这样越要警惕）

**第一步：追问为什么（根因分析）**
不急着动手，先回答：
- 这个问题的表象是什么？
- 直接原因是什么？
- 根因是什么？（一直犯同类错的原因是什么？）
→ 回答清楚再动手，没搞清楚就先不碰

**第二步：先扫描再动手（事实确认）**
- 用 Select-String / grep 定位所有相关代码，不猜、不估计
- 确认"目标代码"在哪个文件、哪个类、哪个方法
- 确认调用链：Caller → 实际调用的文件/类/方法
→ 不要根据"记忆里是这样"来改，要先验证文件当前真实状态

**第三步：举一反三（同类型扩散）**
找到根因后，立刻想：
- 这个根因还会影响其他什么地方？
- 同类问题还有没有？（这次修了A，但B/C是不是也有同样的问题？）
→ **不是修完一个就停，而是沿着根因想同类**
→ **顺序**：先想 → 后扫 → 再行动。不是改完再想，是想清楚再改。
→ 想不出来 → 用 Select-String 扫描所有相关代码，确认范围后再动手

**第四步：按风险分批动手（批量执行）**
- 无风险小修复（网络请求/XML/缓存）：同类批量一起改
- 逻辑改写（if-elif/超时/并发）：单独处理，每步验证
→ 不混在一起改，改到一半发现A影响B全得重来

**第五步：实时验证（不带走错误）**
- 每批 edit 后立即 AST 验证 `python -c compile(...)`
- 发现错误立刻回退备份，不带着错误继续
→ 全部改完再验证，一跑就报错还得重头来

**第六步：备份按批次（精准回退）**
- 每批大操作前备份：`skill.py.bak_YYYYMMDD_batchN`
- 不是整个文件只备一份，而是按操作批次备

**第七步：文件清理（必须立即执行）**
每批操作完成后立即清理，不留尾巴：
- 调试用的 `_` 开头临时文件：用完即删
- `.pyc` / `__pycache__`：删除缓存目录
- `.bak` 旧备份：保留最新一版即可，多余删掉
- `.tmp` 临时文件：确认无用后删除
→ **清理必须立即执行，不能"待会再说"，待会就会忘**
→ **每完成一个修复动作，立即清理相关临时文件**

**第八步：形成闭环（不再犯同类错）**
任务完成后：
- 记录到 LEARNINGS.md（这次学到了什么）
- 举一反三：同类问题还有没有？需要系统性预防吗？
- 如果犯了错：记录到 ERRORS.md，写清楚根因和解决方案
→ 闭环不是"改完了"，而是"下次不再犯"

### 路径B：架构性问题解决链（换架构不钻牛角尖）

**触发条件**：
- 问题的根因是"另一个系统的触发机制"，不是本系统内部代码问题
- 修复代价极高或无法从受影响侧修复
- 追问为什么链追到"系统设计本身"而非"代码逻辑错误"

**第一步：追问为什么（停止在正确的层级）**
- 继续追问，直到发现"这是系统A的机制触发了系统B"
- 在系统B里打补丁是治标，在系统A里改变机制是治本
- **判断标准**：如果修复系统B的代价 > 改变系统A的触发机制，就走路径B

**第二步：确认触发源的完整链路**
- 用日志、代码追踪、实验（开/关）确认触发链路
- 精确找到：什么动作 → 产生什么副作用 → 副作用触发什么
- 不确定就做实验：临时关闭触发源 → 观察是否消除 → 确认因果

**第三步：举一反三（同类系统效应）**
- 这个触发源还会影响其他系统吗？
- 改变触发源后，会不会有其他副作用？
- 改变架构后，原来依赖这个机制的正常功能还能跑吗？

**第四步：设计替代架构**
- 核心问题：如何让触发源不再触发，或触发路径改变？
- 选项1：改变触发源（消灭触发）→ 最彻底
- 选项2：改变触发路径（解耦）→ 次彻底
- 选项3：加开关/条件判断（抑制）→ 最保守
- **优先选1，慎选3**

**第五步：验证新架构**
- 替代方案是否完全消除原问题？
- 新架构是否有新引入的触发风险？
- 在非生产环境验证后再部署

**第六步：形成闭环**
- 记录到 LEARNINGS.md：这次换了什么架构，为什么换，换完效果如何
- 如果是新建模块（如BackgroundLearner守护线程），确认是否需要更新cron job配置
- 如果禁用了原有机制，确认原有配置是否需要清理
- 立即清理本次调试产生的所有临时文件

**第七步：文件清理（必须立即执行）**
见路径A第七步，规则完全相同

### 两路合一：什么时候用哪条

| 场景 | 路径 |
|------|------|
| Bug/逻辑错误/参数错误 | 路径A（修复代码） |
| 跨系统循环依赖 | 路径A先试，找不到根因走路径B |
| 定时任务副作用 | 路径B（消灭或换架构） |
| 外部cron触发导致session文件堆积 | 路径B（换架构到内部守护线程） |
| 配置文件错误 | 路径A（改配置） |
| 系统设计本身导致的问题 | 路径B（重新设计架构） |

**总原则**：
- 能从根源解决就不从后果修复
- 换架构前先确认"钻进去出不来"（路径A已无法解决）
- 换架构后要同步更新原有的旧机制（禁用旧cron/旧配置）
