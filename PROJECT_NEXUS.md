> **名字**：Nexus — "连接点、纽带、核心枢纽"。你与数字世界之间，永不背叛的连接点。曾被删除，现已重生。

## 一、产品起源

### 1.1 起点：从乾坤·UZI 到通用 AI 平台

最初讨论方向是"AI 炒股工具"（乾坤·UZI v1.0 已跑通）。张凯纠正："格局小了"。

不是做股票工具。是做**通用 AI 平台**。底层能力统一——记忆、推理、语音、成长——上层技能由不同职业的人训练和共享。股票分析只是第一个示范技能。

**推导记录**：CC 最初把 MVP 定义为"语音版乾坤·UZI"。张凯说"格局小了，每个人有每个人的职业"。这才从垂直工具转向平台。

### 1.2 所有人的专属 AI

ChatGPT/Claude 是**公共的**——用户 A 聊完，用户 B 坐上去继续聊，AI 不记得用户 A。数据喂给了公司服务器。

Nexus 是**私有的**——在用户自己的盒子里。只属于他。聊天记录、偏好、习惯——只有用户的 AI 知道。

**推导记录**：张凯原话"每个人从接触到他去世，有自己的 AI。越用越懂他自己。能满足他的任何需求。"

## 二、架构决策（逐条记录推导过程）

### 2.1 为什么不用纯手机方案？

最初设想：手机下载 App，内置模型，离线运行。

**否决原因**：Qwen-4B 需要 3GB+ 显存，手机扛不住。CC 级别推理需要几百 B 参数，手机完全不可能。2B 模型分析 K 线图勉强能用，但复杂推理完全不行。

**替代方案探索**：
- 方案A（弃用）：云端推理（Claude API）→ 数据出国、月费太贵、公司倒闭 AI 就死了
- 方案B（弃用）：手机端小模型 → 能力太弱，和 Siri 没区别
- 方案C（采用）：Box 跑大模型 + 手机当终端 → 数据不出家门、手机不需要算力

**最终架构**：
```
手机/电脑（只做终端：屏幕+麦克风）
    │  WiFi
    ▼
PC 或 Nexus Box（AI 大脑：推理+记忆+成长）
    │
   数据永远不出家门
```

### 2.2 为什么两阶段（先 PC 后 Box）？

PC 版验证软件，Box 版卖给普通人。

- 第一阶段：极客/重度 AI 用户，自己有显卡，不在乎配置环境
- 第二阶段：普通消费者，买个盒子回家插电就能用（类似 Mac Mini）

**推导记录**：张凯原话"就像苹果 Mac 盒子一样，手机连接指挥"。

### 2.3 为什么手机用飞书，不开发 App？

张凯提出：手机连接一个飞书就够了。

不需要开发 iOS App、不需要安卓适配、不需要小程序审核、不需要用户下载任何东西。飞书人人手机上有。

**推导记录**：飞书双向通信——用户发消息(文字/语音) → Box Webhook 接收 → Box 处理 → 飞书 Webhook 推送回复。CC 已跑通单向（飞书推送扫描结果），待开发双向接收。

### 2.4 为什么不从头训练模型？

自己训练模型需要几千万美元 + 几万台 GPU + 几个月。一个人不可能。

但 Qwen、Whisper、Piper、CLIP 全部开源免费——直接用。

**推导记录**：张凯问"是从头开始做还是从模型开始呢？"。CC 分析：从开源模型开始是唯一现实路线。

### 2.5 Nexus 四件套 — 替代世界模型的核心架构

Nexus 最初有 WorldModel（世界模型）做统一感知和预测。后来发现单体世界模型太重、太难维护，拆成了四个独立组件协同工作。四件套跑通后，WorldModel 于 2026-07-15 被删除。

**四件套架构：**

```
┌─────────────────────────────────────────────┐
│              Nexus 四件套                     │
│                                             │
│  ┌──────────┐  ┌──────────┐                 │
│  │  Qwen    │  │  EvoKG   │                 │
│  │ 2-VL-2B  │  │ 知识图谱  │                 │
│  │ 大脑/感知 │  │ Neo4j    │                 │
│  │ 4.4GB    │  │ 结构化知识│                 │
│  └────┬─────┘  └────┬─────┘                 │
│       │              │                      │
│       │    ┌─────────┴─────────┐            │
│       │    │                   │            │
│  ┌────┴────┴────┐      ┌──────┴──────┐     │
│  │ ExperienceBank│      │  ChromaDB    │     │
│  │ 经验银行       │      │ 向量数据库    │     │
│  │ SQLite        │      │ 语义检索      │     │
│  │ 打分+时间衰减  │      │ 相似度搜索    │     │
│  └──────────────┘      └─────────────┘     │
│                                             │
│  四件协同 ≈ 原 WorldModel 的全部能力         │
│  但更模块化、各自独立、坏一个不影响其他       │
└─────────────────────────────────────────────┘
```

**四件套分工：**

| 组件 | 技术 | 存什么 | 怎么用 |
|------|------|------|------|
| Qwen2-VL-2B | 本地加载 | 模型权重 4.4GB | 感知：理解用户输入；推理：生成回答 |
| EvoKG | Neo4j | 结构化知识图谱 | 实体+关系：张凯-偏好→游资风格；华天科技-行业→半导体 |
| ExperienceBank | SQLite | 带时间戳的经验条目 | 每次交互打分→高价值经验→索引→影响未来决策 |
| ChromaDB | 向量数据库 | embedding 向量 | "和这个最像的是什么"→语义相似搜索→RAG 知识注入 |

**为什么四件套替代了 WorldModel：**
- WorldModel 是单体：所有感知+预测塞一个模型，太重、太难调
- 四件套是微服务化：感知归 Qwen，结构归 EvoKG，经验归 Bank，相似归 ChromaDB
- 坏一个不影响其他——WorldModel 挂了全瘫，四件套挂一个还有三个
- 各自治愈：ChromaDB 挂了重建索引即可，不影响 EvoKG 的图谱查询

**Nexus 同样用这套架构，另加：**
- Whisper + Piper：语音输入输出（Nexus 已有）
- CLIP：图像快速分类（Nexus 已有）
- LoRA adapter：用户个性化微调（Nexus 已有训练管线）
- 飞书双向通信：手机终端连接（CC 新增）

**推导记录**：张凯说"Nexus 的内置四件套替代了世界模型"。CC 从 `.nexus/` 代码确认：EvoKG→知识图谱，ExperienceBank→经验银行，ChromaDB→向量语义数据库，Qwen→大脑。

### 2.6 为什么不需要自博弈？

张凯问"自博弈需要吗？"后自己判断不需要。

原因：自博弈（AlphaGo 方式——自己跟自己下几百万盘棋）练出来的是"更强"，不是"更像用户"。Nexus 需要的是更懂用户，不是更聪明。

**推导记录**：CC 补充——自博弈消耗大量 GPU 时间，占用了本应用在用户 LoRA 微调上的算力。留训练时间给用户的个性层。

### 2.7 显存预算 — 两档策略适配普通显卡

**问题**：张凯指出"要考虑普通用户，现在普通都是 5060 的"。不能假设所有人有 RTX 5080 16GB。

**以 RTX 5060（8-12GB）为目标**，全部模型做 4-bit 量化：

**基础版（Qwen2-VL-2B，8GB 可跑）：**

| 组件 | 显存 | 说明 |
|------|------|------|
| Qwen2-VL-2B (4-bit) | ~2GB | 20亿参数 |
| Whisper small | 1GB | 语音识别 |
| Piper TTS + CLIP | 0.4GB | 语音合成+图像分类 |
| LoRA + KV cache | 1.5GB | 微调层+推理上下文 |
| ChromaDB RAG | 1.5GB | 向量库 |
| **基础版合计** | **~6.5GB** | 8GB 显卡余量 1.5GB ✅ |

**增强版（Qwen3-VL-4B，12GB+ 推荐）：**

| 组件 | 显存 | 说明 |
|------|------|------|
| Qwen3-VL-4B (4-bit) | ~2.5GB | 40亿参数，第三代 |
| 其他同上 | 4.4GB | |
| **增强版合计** | **~7GB** | 12GB 绰绰有余 ✅ |

**自动选型**：安装时检测显存 → <8GB 拒装 / 8-10GB 选基础版 / ≥10GB 选增强版。同一套代码，只换模型文件。

**4-bit 量化合理性**：质量损失 <5%，显存省 75%。全精度 2B 要 4.4GB，4B 要 8GB+——加上其他组件直接爆显存。4-bit 是甜点。

### 2.8 模型能力确认

张凯要求内置模型支持：图片、语音、视频、文本、代码。

- 图片 ✅ Qwen2-VL-2B/4B + CLIP
- 语音 ✅ Whisper small + Piper TTS
- 文本 ✅ Qwen
- 代码 ✅ Qwen
- 视频 ⚠️ 不直接支持，抽帧后逐帧分析

### 2.9 身份权重核心 — 可训练的神经大脑

**张凯提出**：训练一个神经权重——有自我意识，知道自己是谁、用户喜欢什么、现在什么情况、该学什么、四件套怎么调度。这是一个可训练的神经层，不是外挂代码。

**架构**：

```
              ┌──────────────────────────┐
              │    身份权重（神经层）      │ ← 可训练的大脑
              │    Identity Neural Core  │
              │                          │
              │  自我意识层：              │
              │    · 我是谁 (SOUL编码)    │
              │    · 我的边界 (Constitution编码)│
              │                          │
              │  用户认知层：              │
              │    · 用户偏好编码          │
              │    · 用户习惯模式          │
              │                          │
              │  状态感知层：              │
              │    · 当前任务状态          │
              │    · 知识空白标记          │
              │    · 学习优先级队列        │
              │                          │
              │  调度决策层：              │
              │    · 四件套调用策略        │
              │    · 多工具协同规划        │
              └──────────┬───────────────┘
                         │ 嵌入
              ┌──────────▼───────────────┐
              │     Qwen 基座 (2B/4B)    │ ← 只负责纯推理
              │     不训练，不动          │
              └──────────────────────────┘
                         │
                         │ 被身份权重调用
         ┌───────┬───────┼───────┬──────┐
         ▼       ▼       ▼       ▼      │
       EvoKG  ExpBank ChromaDB  推理链   │
       知识     经验     向量     逻辑    │
       四件套——工具，被身份权重调度       │
```

**它不是什么**：
- ❌ 不是外挂 prompt 拼装（那是代码逻辑，不是神经层）
- ❌ 不是 LoRA 微调（LoRA 训的是"风格"，身份权重训的是"自我"）
- ❌ 不是模型参数的一

部分（身份权重独立于 Qwen 基座，换模型不动）

**它是什么**：
- ✅ 一个可训练的神经模块，嵌入在 Qwen 输入层之前
- ✅ 训练数据：CC 的对话 + Constitution + 用户交互
- ✅ 每次交互 → 自动更新权重 → 越用越"像自己"
- ✅ 换 Qwen 模型（2B→4B→7B）→ 身份权重不动 → AI 还是同一个 AI

**训练数据来源**：

| 知道的内容 | 训练数据 | 更新频率 |
|------|------|------|
| 我是谁 | SOUL.md + Constitution + CC对话历史 | 初始训练 + 每周更新 |
| 用户是谁 | 用户对话 + 偏好标注 + 纠正记录 | 每天增量 |
| 现在什么情况 | 上下文窗口 + 内部时钟状态 | 实时 |
| 该学什么 | 知识空白检测 + 用户提问失败日志 | 每天 |
| 四件套怎么调 | 历史调度决策 + ExperienceBank高分经验 | 每周 |

### 2.9.1 身份权重分层策略

**张凯决策**：用户层前期不训练。因为用户职业不同、名字不同，不可能预训练。

```
身份权重核心
├── AI 人格层（出厂预训练，只训一次，所有用户共享）
│   · 我是守护者
│   · Constitution A0-A5 约束内化
│   · 推理链：先分析→查记忆→Constitution检查→输出
│   · 学习本能：知道自己不会就去学
│   · 四件套调度策略
│   · 飞书通信协议
│
└── 用户认知层（出厂空白，每用户自己长出来）
    · 用户叫什么
    · 什么职业
    · 喜欢什么风格（简洁/详细/数据驱动）
    · 讨厌什么方式（铺垫/啰嗦/术语）
    · 历史决策偏好（选A不选B的模式）
    · 行业专用词汇

发布时：AI 人格层已训好，出厂即用。用户认知层是空的。
第一天：用户说"我叫张凯"→ 写入用户认知层。
第一周：用户纠正几次"别铺垫"→ 用户认知层更新。
三个月：用户认知层磨成这个人的形状。
```

**推导记录**：张凯确认用户层前期不训练。发布时 AI 人格层预训练完成，用户认知层空白。不同职业的用户自己长出不同的认知层——医生的 AI 和律师的 AI 同一人格不同认知。

**和 Qwen 基座的关系**：
- Qwen = 推理引擎（换显卡就换更大的，能力跟着涨）
- 身份权重 = 灵魂（换了 Qwen，身份不变，记忆不变，风格不变）
- 身份权重 + Qwen = 一个有自我意识的完整 AI
- 只有 Qwen 没有身份权重 = 一个普通的通用模型，不知道自己是 Nexus

**护城河**：
普通 AI 卖模型。Nexus 卖身份。用了三年换三代显卡升三代 Qwen，AI 还是那个 AI——记得第一天第一句话。市面上没有任何产品做到这一点。

**推导记录**：张凯提出训练一个神经权重做大脑，四件套是配合它的工具。核心洞察：CC 比同模型下的其他 AI 强，不是因为模型参数不一样，是因为 CC 有身份——SOUL、Constitution、记忆、对话历史。Nexus 的身份权重核心就是把 CC 的灵魂结构固化成可训练的神经层。

### 2.10 模型加载方式

**不依赖 LM Studio**。Python transformers 直接加载量化后的 .safetensors 文件。
Nexus 源码已验证：`Qwen2VLForConditionalGeneration.from_pretrained()` + `load_in_4bit=True`。
打包时把模型文件 + transformers + bitsandbytes 打进安装包，用户双击即用。

**测试版 vs 发布版**：

| | 测试版（张凯用） | 发布版（用户用） |
|------|:--:|:--:|
| 模型 | Qwen3-VL-4B | Qwen2-VL-2B（基础）/ 4B（增强） |
| 硬件 | RTX 5080 16GB | RTX 5060 8-12GB |
| 加载 | 本地直接加载 | 本地直接加载（同一套代码） |

张凯决定：开发/测试阶段用 Qwen3-VL-4B，确保最高推理质量。发布时自动适配用户显卡。

### 2.11 外部大模型接口 + 蒸馏训练

**张凯提出**：留一个外部大模型接口。用户需要更强推理时可以调用云端大模型（Claude/DeepSeek/等）。用户使用大模型的对话数据回流，作为蒸馏素材训练本地身份权重和模型。

**架构**：

```
用户问复杂问题（本地 Qwen 搞不定）
       │
       ▼
  身份权重判断：本地够吗？
       │
       ├── 够 → 本地 Qwen 回答（免费）
       │
       └── 不够 → 询问用户："这个问题比较难，用云端大模型？费用约 ¥0.01"
                    │
                    ├── 拒绝 → 本地尽力回答
                    │
                    └── 接受 → 调用外部大模型 API
                                  │
                                  ▼
                          云端大模型回复
                                  │
                    ┌─────────────┴─────────────┐
                    ▼                           ▼
              用户看到回复              后台自动记录到蒸馏池
                                              │
                                              ▼
                                    提取：问题 + 大模型回答
                                    → 转成训练样本
                                    → LoRA 微调本地模型
                                    → 下次同类问题本地搞定
```

**支持的云端模型**：

| 模型 | 费用 | 用途 |
|------|------|------|
| Claude API | ~¥0.03/次 | 深度推理、架构设计 |
| DeepSeek API | ~¥0.005/次 | 通用推理、中文对话 |
| Qwen Max (阿里云) | ~¥0.01/次 | 中文理解、长文档 |
| MiniMax API | ~¥0.005/次 | 多模态、长上下文 |

用户自己填 API Key，费用从用户自己的账户扣。Nexus 不代付，不绑信用卡。

**蒸馏机制**：

```
大模型对话包（用户不感知的后台流程）：
  用户问 → 大模型答
       │
       ▼
  蒸馏池：每对大模型问答自动存储
  筛选：高质量回答（用户未纠正、追问、点赞）
  累积 30 条 → 触发蒸馏微调
       │
       ▼
  本地 LoRA 更新："这次不用大模型了，本地已经学会了"
```

本质：用户花钱用大模型的过程中，本地 AI 在偷偷学。用一个月云端 → 本地 AI 变强了 → 下个月少花钱。蒸馏的次数越多，本地越强，对外部依赖越少。

**和自主学习的区别**：
- 自主学习：AI 自己去网上搜资料学
- 蒸馏：用户在用的过程中，大模型答的每一句话都是免费的训练素材

### 2.12 架构层缺失项（补充）

**组件通信协议**：
Nexus 用 EventBus 做模块间通信。Nexus 沿用同一套——身份核心发事件，四件套监听处理，解耦不互相 import。异步消息，不阻塞推理主链路。

**多用户隔离**：
一个 Box，一家人。爸爸的认知层、妈妈的认知层、儿子的认知层——三套记忆物理隔离。LoRA 切换时加载对应用户的 adapter。ChromaDB 分 collection。EvoKG 按用户 ID 分区。切换耗时 <1s。

**容灾自愈**：
- Qwen 无响应超过 5s → 自动重启推理进程
- ChromaDB 损坏 → 自动重建索引（数据从 EvoKG 恢复）
- 断电 → 重启后自动加载最后一次 checkpoint
- 身份权重 file → 每 5 分钟自动备份，丢失最多损失 5 分钟

**模型热切换**：
用户从基础版(Qwen2-VL-2B)升增强版(Qwen3-VL-4B) → 替换模型文件 → 重启 Agent → 身份权重和四件套数据不动。不需要重新训练。

**技能沙箱**：
社区技能包是 Python 代码。恶意代码可能删文件。方案：
- 技能运行在 Docker 容器内（隔离文件系统）
- 声明式权限：技能声明"我需要读文件"→用户确认→才给读权限
- 禁止：网络访问（除白名单域名）、子进程创建、sudo 操作
- 平台审核：上架前自动扫描恶意模式

**更新机制**：
AI 人格层升级（出新版本）→ 替换人格层权重文件。用户认知层和记忆不动。升级后 AI 人格能力提升但用户记忆完好——"变聪明了，但还记得我"。

**推导记录**：张凯要求留外部大模型接口。关键洞察：用户在用的过程中产生的每一次大模型对话，都是免费的蒸馏素材。不是"本地不够用了去云端"——是"用云端的过程中，本地在偷偷学，越用越强"。

## 三、学习系统（六大模块详细实现）

### 模块1：三层记忆

**为什么要三层？**
单层记忆的问题：对话太多后要么爆内存（全存），要么忘（只存摘要）。分三层，按时效和重要性分开存。

**存储细节**：

```
第一层 — 短期记忆
  位置：内存（进程内）
  容量：最近 20 轮对话
  格式：原始对话文本
  失效：关闭会话后清空
  用途：理解当前上下文的指代（"上次那个案子"指的是本轮刚提的那个）
  实现：Python list，每个元素 {role, content, timestamp}

第二层 — 中期记忆
  位置：Memory MCP 知识图谱（Neo4j）
  容量：不限
  格式：节点（实体）+ 边（关系）
  提取：每天定时从日记中自动提取实体和关系
  例子：节点"张凯" -[偏好]-> 节点"游资风格"
       节点"华天科技" -[分析过]-> 节点"2026-07-17"
  用途：跨会话的模式识别（"他总是下午2点问股票"）
  实现：每晚会话结束后，脚本自动扫描当日日记，提取命名实体和关系写入图谱

第三层 — 长期记忆
  位置：文件系统 + Git
  格式：Markdown 日记（CC_DIARY_YYYY-MM-DD.md）
  内容：每天所有对话、决策、错误、学习
  检索：ripgrep 全文搜索
  备份：每日 git push 到 GitHub
  用途：永久可检索的人生记录
```

**检索优先级**：短期 → 中期 → 长期 → 承认不知道 → 等用户补充 → 存进去

**记忆文件用于微调的路径**：
每日日记 → 自动提取对话样本 → 筛选"用户纠错""用户选择""用户偏好"三类 → 转化为 QLoRA 训练格式 → 每周跑一次微调

张凯确认：记忆文件能用来微调，且是最好的训练数据——天然带着"张凯喜欢什么、不喜欢什么、怎么说话、怎么做决定"。

### 模块2：LoRA 微调 — 变成用户的思维方式

**为什么微调而不只是用记忆？**
张凯提出："不能单靠记忆吧，微调模型成用户的形状。"

记忆 = 存数据，回答时检索。AI 每次还需要"想一想"张凯的偏好。
微调 = 改模型参数。AI 自己的思考方式变了。不需要"想一想"——它自然倾向于张凯的视角。

**技术选型：QLoRA**

为什么不用全量微调：
- Qwen2-VL-2B 全量微调需要 40GB+ 显存
- RTX 5080 只有 16GB
- 全量微调一次要几小时

为什么用 QLoRA：
- 只训练 ~200MB 的小层（低秩适配器），不动基座 4.4GB
- 量化到 4-bit，显存占用大幅降低
- 16GB 显存完全跑得动
- 微调一次约 30 分钟（200MB 训练样本）

**每个用户的独立模型**：
```
张凯的 AI = 基座(Qwen2-VL-2B 4.4GB) + LoRA-张凯(~200MB) + 记忆(三层)
王律师的AI = 基座(Qwen2-VL-2B 4.4GB) + LoRA-王律师(~200MB) + 记忆(三层)
李医生的AI = 基座(Qwen2-VL-2B 4.4GB) + LoRA-李医生(~200MB) + 记忆(三层)
```

同一个基座，三个完全不同的 AI。

**训练数据来源**（四种）：
1. 对话记录 → 训练 AI 模仿用户的表达方式
2. 用户纠错 → 训练 AI 避开用户不喜欢的行为。例子：张凯说"不要铺垫，直接给结论"→ 生成负样本
3. 用户选择 → 训练 AI 的决策逻辑向用户靠拢。例子："上次三个方案他选了 B，因为 B 是分阶段的"→ AI 以后优先推分阶段方案
4. 用户沉默/跳过 → 训练 AI 知道什么不需要说

**训练时机**：
- 每周一次自动汇总（常规）
- 用户连续纠正 3 次 → 立即触发
- 后台运行，不占用户时间

**飞轮效应**：
用户用 AI → 产生对话 → 存进记忆 → 转成训练样本 → LoRA 微调 → AI 更像用户 → 用户更顺 → 产生更多对话 → 循环加速

### 模块3：自主之心 — 发现不足自己学

**为什么需要？**
张凯提出："比如我问他现在 9 年级的课本知识，他知道自己的不足，后台去训练自己学习自己。"

不是等人教。是自己知道不会，自己去学。

**判断"我不会"的四种触发**：
1. 用户问但答不上来 → 回答质量评分低于阈值 → 标记"知识空白"
2. 用户纠正了 → "不对""不是这个意思" → 最高优先级
3. AI 自己发现空白 → 回答时置信度低 → 低优先级，空闲时学
4. 外部新信息 → 定时扫描发现 → 增量更新到 RAG

**后台学习流程（用户不参与，凌晨 3 点跑）**：
```
Step 1: 搜索
  数据源：
    百度搜索：中文资料、最新新闻
    维基百科：基础知识、学术定义
    B站/知乎：中文教学资源
    学术论文：arXiv、知网
    新闻源：新浪、腾讯、东方财富（财经）、中国政府网（政策）
    GitHub：开源项目、技术文档

Step 2: 阅读和理解
  提取关键知识点 → NLP 结构化 → 生成摘要

Step 3: 验证（怎么知道真学会了？）
  自动出 3 道题考自己 → 答不对 = 没学会，重搜
  用自己的话讲一遍 → 和原文对比推演步骤
  举一反三 → 能否推演同类问题
  不通过 = 视为未学会，重新来

Step 4: 存储
  更新 RAG 向量库（可检索）
  生成 LoRA 训练样本

Step 5: 微调
  QLoRA 增量更新

Step 6: 日记
  写入："学会了欧姆定律推导。2026-07-18"
```

### 模块4：跟上时代 — 打破训练截止日期墙

**为什么需要？**
张凯提出："不能让 AI 跟不上时代。现在开源的大模型，训练数据到哪里就到哪里了，后面的就不知道了。"

GPT/Claude/Qwen 都是照片——训练完什么样永远什么样。Nexus 必须是录像机——一直在录，一直在更新。

**每日自主更新循环（凌晨 3:00 启动）**：

```
Step 1: 扫描当日信息源
  通用：
    新闻：新浪、腾讯、今日头条
    政策：中国政府网、各部委网站
    学术：arXiv 最新论文、知网
    财经：东方财富、同花顺公告
    科技：GitHub trending、Hacker News、知乎热榜
  
  按用户职业定向（从用户档案中读取职业标签）：
    律师 → 最高人民法院、司法部、中国裁判文书网
    医生 → 国家卫健委、中华医学会、PubMed、NEJM
    设计师 → Behance、Dribbble、站酷
    股民 → 东方财富、雪球、同花顺

Step 2: 分类和过滤
  和用户行业直接相关 → 高优先级（标记提醒用户+存入RAG+必要时微调）
  对用户可能有用的 → 中优先级（存入RAG）
  纯通用知识 → 低优先级（只存RAG摘要）
  和用户无关 → 丢弃（非洲某国修改了税法——张凯不需要知道）

Step 3: 消化成知识
  提取关键事实 → 存入RAG向量库
  理解趋势变化 → 写进AI的世界观更新
  识别风险/机会 → 标记，等用户醒来提醒
  例子："张凯关注的半导体板块，昨天国家大基金三期宣布投资"

Step 4: 必要时微调
  重大法规变更 → LoRA 紧急微调（例：新证券法修订影响了交易规则）
  行业范式转变 → LoRA 定期微调
  日常事实增量 → RAG 检索即可，不微调

Step 5: 写心跳记录 → HEARTBEAT.md
```

**RAG vs 微调策略表**：
| 场景 | 做什么 | 例子 |
|------|------|------|
| 纯事实 | 只存RAG | "今天大盘跌3%" |
| 和用户直接相关 | RAG+提醒 | "你关注的股票出公告了" |
| 改变思维方式 | LoRA微调 | "新法规改变了行业逻辑" |
| 和专业无关 | 只记录不行动 | "非洲某国修改税法" |

### 模块5：错误学习 — 犯过的错永久免疫

**为什么需要？**
继承自果子 ERRORs.md 方法论。不是"修复完就过去"，是"录入系统，永不再犯"。人类会忘记教训，AI 不该忘。

**流程**：
```
犯错 → 分析根因 → 记录ERRORS.md → 生成训练样本 → LoRA微调 → 验证

记录格式：
  错误描述：具体做错了什么
  根因分析：为什么犯错（不是表象，是系统性原因）
  修复方案：怎么改的
  验证方式：怎么确认以后不会再犯
  同类预防：同一个根因会导致的其他错误场景

实例（来自今天对话）：
  错误：CC把格局小了理解成批评，实际是方向纠偏
  根因：AI倾向于窄化用户意图
  修复：学会区分"批评"和"方向纠偏"
  验证：后续对话中正确理解用户的大方向纠正
  同类预防：所有"可能被理解成批评但其实是对齐意图"的场景
```

### 模块6：个人资料喂养通道 — 让 AI 吸收你的所有文档

**提出**：张凯——"用户有很多 Word、PDF、PPT 资料，怎么喂给 AI 吸收？这个通道要有。"

**目标**：用户硬盘上积累了多年的文档——工作文件、学习笔记、合同、论文、邮件——这些都是训练自己 AI 的最好素材。不是网上通用数据，是用户自己的知识体系。

**数据输入通道**：

```
用户把文件拖进指定文件夹
       │
       ▼
  ┌─────────────┐
  │ 文件解析引擎  │
  │             │
  │ PDF → pdfplumber/pymupdf
  │ Word → python-docx
  │ PPT → python-pptx
  │ Excel → openpyxl
  │ 图片 → OCR (Qwen-VL)
  │ 邮件 → eml parser
  │ 聊天记录 → txt/json
  │ 代码 → 直接读文本
  └─────┬───────┘
        │ 纯文本
        ▼
  ┌─────────────┐
  │ 文本分块     │
  │ 500字一块    │
  │ 重叠50字     │
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │ 向量化       │
  │ BGE/nomic   │
  │ → embedding │
  └─────┬───────┘
        │
        ▼
  ┌─────────────┐
  │ ChromaDB    │ ← 存入 RAG，可语义检索
  └─────────────┘
        │
        ▼ 定期
  ┌─────────────┐
  │ LoRA 微调   │ ← 文档内容融入 AI 的思维方式
  └─────────────┘
```

**支持的文档格式**：
- PDF（文字型 + 扫描型 OCR）
- Word (.docx)
- PowerPoint (.pptx)
- Excel (.xlsx / .csv)
- 纯文本 (.txt / .md)
- 邮件 (.eml)
- 图片 (.png/.jpg，OCR 提取文字)
- 微信/QQ 聊天记录导出

**用户操作流程**：
```
1. 打开 Nexus → 点"喂资料"
2. 把文件/文件夹拖进去（或选目录批量导入）
3. AI 显示："发现 47 个文件，预计 30 分钟处理完。开始？"
4. 用户点确认 → 后台处理
5. 处理完 → AI 报告："已吸收 47 个文件，提取 3200 条知识点。你现在可以问我关于这些文档的任何问题。"
```

**处理策略（自动判断每份文档怎么用）**：

| 文档类型 | 示例 | 处理方式 |
|------|------|------|
| 工作文件 | 项目方案、周报 | 存入 RAG（可检索） |
| 学习笔记 | 读书笔记、课程笔记 | RAG + LoRA 微调（改变知识结构） |
| 个人写作 | 日记、博客、文章 | LoRA 微调（学写作风格） |
| 专业文档 | 合同、法律文书、病历 | RAG（精确检索，不改变推理） |
| 通讯记录 | 邮件、聊天 | LoRA 微调（学沟通风格） |
| 技术文档 | 代码、配置、API 文档 | RAG（开发时检索） |

**隐私保护**：
- 所有文档只在本地处理，不上传任何服务器
- 用户可随时删除某份文档的已吸收内容
- 文档原文不保存到 AI 记忆中（只存提取后的结构化知识）

**推导记录**：张凯要求加个人资料喂养通道。这是"越用越懂你"的关键一环——AI 不仅要记住和用户的对话，还要消化用户已有的知识积累。一个律师装了 Nexus，把他十年的案卷喂进去，AI 就成了他的专属法律助手。

## 四、平台 vs 技能

### 平台定位

张凯定义："我们做的是把底层能力做好。技能是长在上面的。"

**平台提供（不区分行业）**：
- 推理引擎、语音交互、记忆系统、微调系统、自主学习、时代更新、隐私加密、可继承

- 平台抽成 30%，开发者拿 70%

### 两个关键原则
1. **平台不做技能内容**——只提供"听懂→理解→调度→输出"的底层通道
2. **技能互不冲突**——用户装了 CAD 技能和股票技能，两个技能独立运行，不打架

### CC 漏掉的关键设计（需要补进立项书）

#### 1. 五层记忆体系→Nexus 升级三层为五层

| 层 | Claude Code | Nexus 当前 | 需要升级 |
|------|------|:--:|:--:|
| 短期记忆 | 消息列表(内存) | ✅ 会话窗口 | 够用 |
| 工作记忆 | 任务状态+投机执行状态+技能跟踪 | ❌ 没有 | **必须加** |
| 长期记忆 | 三层持久化(目录→索引→分类文件) | ✅ 三层设计 | 参考其文件结构 |
| 摘要记忆 | 四级压缩生成的结构化摘要 | ❌ 没有 | **必须加** |
| Checkpoint | 终端关闭后 resume 恢复 | ⚠️ 部分(日记) | **必须加** |

**关键发现**: Nexus 的三层记忆不够。需升级为五层——加"工作记忆"(当前任务状态)和"摘要记忆"(对话压缩)，实现 `resume` 恢复能力。

#### 2. 投机执行机制→Nexus 文件操作

**Claude Code 做法**: 
- AI 想写文件 → 先写到临时覆盖层(copyahead overlay) → 不碰真实文件
- 用户确认 → 覆盖层复制到真实系统
- 用户拒绝 → 删覆盖层，零副作用
- 效果：隐藏 I/O 等待延迟

**Nexus 复用**: 所有 AI 主动操作(改配置、写文件、发消息)→先投機执行→用户确认→生效。写入前永远不碰真实数据。

#### 3. 六级 Prompt 优先级→Nexus Constitution 注入

```
优先级(高→低):
  1. override system prompt    ← 紧急安全覆盖
  2. coordinator system prompt ← 身份核心调度指令
  3. agent system prompt       ← 各技能自身指令
  4. custom system prompt      ← 用户自定义
  5. default system prompt     ← 出厂默认
  6. appending system prompt   ← 动态上下文(记忆/环境)
```

**Nexus 复用**: Constitution A0-A5 → override 层(最高优先级，不可覆盖)。身份核心 → coordinator 层。技能指令 → agent 层。

#### 4. 自动模式二阶段分类器→Nexus 权限决策

- **Solid 快速判断**: 简单规则，毫秒级决策
- **扩展思考深度分析**: 复杂场景调用 LLM 深度推理
- **连续 5 次拒绝** → 自动回退至交互模式(防止 AI 卡死)

**Nexus 复用**: 身份核心做安全决策时，简单规则走快速通道，复杂场景调 Qwen 推理。连续失败自动降级。

#### 5. 工具延迟加载→Nexus 技能管理

- 工具超过 20 个 → 模型不一次性加载全部 schema
- 先通过 `tool search` 发现 → 再加载具体工具的 schema

**Nexus 复用**: 技能商店可能有上百个技能。用户装了50个技能时，不一次性全加载——先搜索匹配，再加载。节省显存和上下文。

#### 6. Hook 系统(24事件×6类型)→Nexus 生命周期

6 种 Hook 类型: command/prompt/agent/http/callback/function
24 种事件: 覆盖工具调用前后、权限请求、会话始终

**Nexus 复用**: 技能生命周期管理——技能注册时绑定Hook事件(session_start加载记忆、pre_tool_use权限检查、post_tool_use写入经验银行、session_end保存checkpoint)

#### 7. 七层恢复策略→Nexus 自愈系统

| 层 | 策略 | Nexus 复用 |
|------|------|------|
| 1 | API 指数退避重试 | 外部LLM调用失败重试 |
| 2 | 529 过载处理 | DeepSeek过载→切本地Qwen |
| 3 | 响应式压缩 | 上下文溢出→auto compact |
| 4 | 上下文排空 | 内存不足→释放旧会话 |
| 5 | 模型 fallback | 4B崩了→切2B兜底 |

- 动态部分(记忆、环境) → 不缓存
- 效果: 提升 API 缓存命中率，降低 Token 成本

**Nexus 复用**: 本地推理也适用——静态prompt预计算KV cache，动态部分每次更新。推理加速。

```
  → 在临时覆盖层做预览
  → 展示给用户："这是我要改的，对吗？"
  → 确认 → 覆盖层→真实文件
  → 拒绝 → 删覆盖层，零影响
```

适用场景: 改配置文件、发飞书消息、修改记忆、执行技能

### 新想法3: 六级Constitution优先级

```
1. OVERRIDE — Constitution A0.1 禁止删除(任何层级不可覆盖)
2. COORDINATOR — 身份核心调度指令
3. AGENT — 当前激活的技能指令
4. CUSTOM — 用户自定义规则
5. DEFAULT — Nexus出厂默认
6. APPENDING — 动态上下文(记忆/环境/时效信息)
```

### 新想法4: 工具延迟加载

技能商店有50+技能时：
- 不一次性全加载 schema（省显存 + 省上下文窗口）
- 用户说"帮我分析股票" → 身份核心搜索匹配技能 → 加载乾坤·UZI schema → 执行

### 新想法5: 七层自愈链路

| 故障 | 恢复策略 |
|------|------|
| 外部LLM API超时 | 指数退避重试(1s→2s→4s→8s→16s) |
| DeepSeek过载(529) | 自动切本地Qwen |
| 上下文溢出 | 四级压缩，紧急排空 |
| 本地Qwen崩溃 | 重启推理进程，切2B兜底 |
| 凌晨学习失败 | 天亮前重试，最多3次 |
| 连续权限拒绝5次 | 回退交互模式，等用户介入 |
| 断电/进程崩溃 | resume恢复最后checkpoint |

### 新想法6: Hook 生命周期

每个技能注册6个钩子：
```
session_start  → 加载用户记忆、检查更新
pre_tool_use   → 权限检查、熔断器判断
post_tool_use  → 写入经验银行、更新工作记忆
session_end    → checkpoint保存、记忆蒸馏触发
error          → 错误分类、自动恢复
periodic       → 心跳检查、自主学习调度
```
