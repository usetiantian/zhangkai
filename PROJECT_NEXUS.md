## 一、产品定义

**每个人的专属 AI，越用越懂他，能在后台自己学习成长，永远跟上时代。**

### 差异化定位

| | ChatGPT/Claude | Siri/小爱 | **NexusAgent** |
|------|:--:|:--:|:--:|
| 归属 | 公司 | 公司 | **用户的** |
| 数据 | 上传云端 | 上传云端 | **本地（不出家门）** |
| 记忆 | 遗忘 | 遗忘 | **三层记忆，越用越懂** |
| 能力 | 固定 | 固定 | **技能商店，无限扩展** |
| 成长 | 等公司更新 | 等苹果更新 | **自己学，每天比昨天强** |
| 时效 | 训练截止日 | 训练截止日 | **每天扫描当天信息，跟上时代** |
| 付费 | 月租 | 免费 | **买断制** |
| 继承 | 不支持 | 不支持 | **数字遗产，传给下一代** |

### 对比总结
- ChatGPT 是**照片**——训练完什么样永远什么样
- NexusAgent 是**录像机**——一直在录，一直在更新，一直在成长

## 二、架构设计

### 部署架构

```
┌──────────┐      WiFi       ┌──────────────────┐
│ 手机/电脑  │◄──────────────►│ PC 或 Nexus Box   │
│ 只做终端  │  语音输入/输出  │                  │
│ 屏幕+麦   │               │ AI 大脑           │
│          │               │ 记忆/推理/成长     │
└──────────┘               │ 数据不出家门      │
                           └──────────────────┘
```

### 两阶段路线

| 阶段 | 产品形态 | 硬件 | 用户 |
|------|---------|------|------|
| 第一阶段 | PC 软件版 | 用户自有电脑+中端显卡 | 极客/重度AI用户 |
| 第二阶段 | Nexus Box | 专用硬件（如 Mac Mini） | 普通消费者 |

### 手机连接：飞书，不开发 App

- 用户手机上的飞书 = 终端界面
- 飞书双向通信：用户发消息(文字/语音) → Box 接收处理 → 飞书推送回复
- 不开发 App，不审小程序，用户不需要下载任何东西
- 已跑通：飞书推送扫描结果（单向）→ 待开发：接收用户消息（双向）

### 模型架构：复用 Nexus 已有 6GB 模型包

| 模型 | 大小 | 用途 |
|------|------|------|
| Qwen2-VL-2B | 4.4GB | 视觉+文字+代码——AI 大脑 |
| Whisper small | 1GB | 语音识别——听用户说话 |
| Piper TTS | 63MB | 中文语音合成(huayan)——说给用户听 |
| CLIP | 350MB | 图像理解——识别图片内容 |

总大小 6GB，均为开源免费。不从头训练，不依赖云端 API。

### 每用户额外存储

| 组件 | 大小 | 说明 |
|------|------|------|
| LoRA 微调层 | ~200MB | 用户专属，越用越像用户 |
| 三层记忆 | 不定(累积) | 短期日记+中期图谱+长期Git |
| RAG 知识库 | ~500MB | 向量化存储，实时检索 |

## 三、学习系统（五大模块，缺一不可）

### 模块1：三层记忆 — 记住用户的一切

**目标**：AI 和用户聊完，下次不需要从头问起。

**三层结构**：

```
第一层：短期记忆（对话级）
  存储：当前会话的上下文
  技术：会话窗口，最近 20 轮对话
  例子：用户刚才说"帮我查华天科技"→AI知道这一轮的context

第二层：中期记忆（系统级）
  存储：跨会话的模式和关系
  技术：Memory MCP 知识图谱（Neo4j），节点+关系
  例子：用户总是下午2点问股票 → AI 自动在那个时间准备扫描

第三层：长期记忆（永久级）
  存储：写入文件系统+Git版本控制
  技术：日记(CC_DIARY_*.md) + SOUL.md + Git push
  例子：用户三年前的重大决策 → 永久保存，可搜索
```

**检索流程**：
```
用户说："上次那个案子怎么样了"
  → 先查短期（本轮提到过吗？）
  → 再查中期（MCP图谱里有"案子"相关节点吗？）
  → 最后查长期（日记文件里grep"案子"）
  → 找不到 → 承认不知道 → 等用户补充 → 存入记忆
```

### 模块2：LoRA 微调 — 变成用户的思维方式

**目标**：不是"记住用户喜欢什么"，是"AI 的思考方式变成用户的形状"。

**和记忆的区别**：
- 记忆 = 存数据，回答时检索
- 微调 = 改模型参数，AI 自己的思考倾向改变了

**技术方案**：QLoRA

```
基座模型：Qwen2-VL-2B（4.4GB，只读，不动）
微调层：  LoRA adapter（~200MB，每个用户独立）

训练数据来源：
  1. 对话记录 → 训练AI模仿用户的表达方式
  2. 用户纠错 → 训练AI避开用户不喜欢的行为
     （"不要铺垫""直接给结论""用游资视角"）
  3. 用户选择 → 训练AI的决策逻辑向用户靠拢
     （"上次三个方案他选了B"）
  4. 用户沉默/跳过 → 训练AI知道什么不需要说

训练时机：
  - 每周自动汇总一周对话 → 提取训练样本 → LoRA微调
  - 事件驱动：用户连续纠正3次 → 立即触发训练
  - 后台运行，不占用用户使用时间
```

**个性化效果示例**：
```
张凯的AI              王律师的AI              李医生的AI
─────────            ─────────              ─────────
回答风格：一句话     回答风格：法条引用       回答风格：鉴别诊断
决策倾向：游资视角   决策倾向：风险规避       决策倾向：循证医学
操作方式：全权授权   操作方式：事事确认       操作方式：看情况
```

**飞轮效应**：
```
用户用AI → 产生对话 → 存进记忆 → 转成训练样本
                                    │
                                    ▼
                          LoRA微调更新200MB小层
                                    │
                                    ▼
                AI更像用户 ← 用户更顺 ← 产生更多对话
```

### 模块3：自主之心 — 发现不足自己学

**目标**：不是等人教，是知道自己不会，自己去学。

**判断"我不会"的时刻**：

| 触发类型 | 如何判断 | 处理 |
|------|------|------|
| 用户问但答不上来 | 回答质量评分低于阈值 | 标记"知识空白"，后台学 |
| 用户纠正了 | 用户说"不对""不是这个意思" | 最高优先级，立即学 |
| AI自己发现空白 | 回答时置信度低 | 设为低优先级，空闲时学 |
| 外部新信息 | 定时扫描发现新知识 | 增量更新到RAG库 |

**后台学习流程**（用户不参与）：
```
用户问："九年级物理，欧姆定律怎么推导？"
  → AI尽力回答了，但深度不够
  → AI自己标记：物理知识不足
  → 凌晨3点后台启动：
      1. 搜索：课本教材、公开课视频、维基百科
      2. 阅读理解：提取欧姆定律的推导步骤
      3. 验证：自己能讲清楚吗？（出题考自己）
      4. 存储：更新RAG知识库 + 生成LoRA训练样本
      5. 微调：LoRA增量更新
      6. 日记：记录"学会了欧姆定律推导"
  → 第二天用户再问 → 答得比昨天好
```

**搜索数据源配置**：
- 百度搜索：中文资料、最新新闻
- 维基百科：基础知识、学术定义
- B站/知乎：中文教学资源、专业知识社区
- 学术论文：arXiv、知网
- 新闻源：新浪、腾讯、东方财富（财经）、中国政府网（政策）
- GitHub：开源项目、技术文档

**验证机制**（怎么知道真学会了）：
```
学完后：
  1. 自动出3道题考自己 → 答不对 = 没学会，重新学
  2. 用自己的话讲一遍 → 和原文对比，关键推演步骤对得上
  3. 举一反三：能否推演同类问题
  4. 不出题考核，视为未学会，重新搜资料
```

### 模块4：跟上时代 — 打破训练截止日期的墙

**问题**：Qwen2-VL-2B 的训练数据有截止日期。2024年6月之后的事，它全不知道。新法律、新政策、新科技、昨天的大新闻——一问就瞎猜。

**解法：不等人更新，自己更新自己。**

**每日自主更新循环**（用户睡觉时跑）：

```
凌晨3:00 启动：
  1. 扫描当日信息源
     新闻：新浪、腾讯、今日头条头条新闻
     政策：中国政府网、各部委网站
     学术：arXiv最新论文
     财经：东方财富、同花顺公告
     科技：GitHub trending、Hacker News
     专业：百度学术、知网（按用户职业定向）

  2. 分类和过滤
     和用户行业相关的 → 高优先级
     对用户可能有用的 → 中优先级
     通用知识 → 低优先级（只存RAG）
     和用户无关的 → 丢弃

  3. 消化成知识
     提取关键事实 → 存入RAG向量库（可检索）
     理解趋势变化 → 写进AI的"世界观"更新
     识别风险/机会 → 标记，等用户醒来提醒

  4. 必要时微调
     重大法规变更 → LoRA紧急微调
     行业范式转变 → LoRA定期微调
     日常增量 → RAG检索即可，不微调

  5. 写心跳记录
     写到HEARTBEAT.md：
     "昨日更新：民法典新解释第N条、沪深交易所新规"
     "今日提醒：张凯关注的板块出了新政策"
```

**更新策略（什么情况下做什么）**：

| 怎么更新 | 适用场景 | 例子 |
|------|------|------|
| 只存RAG | 纯事实更新 | 今天大盘跌了3% |
| RAG+提醒 | 和用户直接相关 | 关注的股票出公告了 |
| LoRA微调 | 思维方式需要调整 | 新法规改变了行业逻辑 |
| 只记录不行动 | 和专业无关 | 非洲某国修改了税法 |

### 模块5：错误学习 — 犯过的错绝不犯第二次

> 补充项——果子ERRORS.md的方法论，必须纳入

**触发**：AI做错了、被用户纠正了、结果不符合预期

**流程**：
```
犯错 → 分析根因 → 记录ERRORS.md → 生成训练样本 → LoRA微调 → 验证

记录格式：
  - 错误：把用户说"格局小了"当成批评，实际是纠正方向
  - 根因：AI倾向于窄化用户意图
  - 修复：学会区分"批评"和"方向纠偏"
  - 验证：后续对话中正确理解用户的纠正意图
  - 同类预防：所有"可能被理解成批评但其实是对齐"的场景
```

**核心原则**：不是"修复完就过去"，是"录入系统，永不再犯"。
